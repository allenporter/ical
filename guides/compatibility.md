# Compatibility Guide

The `ical` library strictly adheres to RFC 5545 by default. However, real-world iCalendar (`.ics`) files generated by major calendar providers often contain deviations and malformed data. To handle these gracefully, the library provides a compatibility layer (`ical.compat`) that applies targeted fixups when reading such files.

## Motivation & Pain Points

Strict parsing is beneficial for detecting bugs when generating iCalendar data (e.g. Home Assistant's `local_calendar`), ensuring the system never produces invalid output. However, for consuming external data (e.g., Home Assistant's `remote_calendar`), strict parsing leads to frequent failures.

Real-world examples of integration failures due to strict parsing include:
- **DTSTART and UNTIL Mismatches**: External providers generate feeds where `DTSTART` is a `DATE-TIME` but `UNTIL` is a `DATE`, violating the specification that they must be the same value type ([home-assistant/core#146621](https://github.com/home-assistant/core/issues/146621)).
- **Malformed Property Lines**: iCloud calendars sometimes contain property lines that don't strictly adhere to syntax rules, causing the parser to fail entirely on unexpected delimiters ([home-assistant/core#170999](https://github.com/home-assistant/core/issues/170999)).
- **Custom Timezones**: Microsoft Office 365 and Exchange frequently use proprietary timezone identifiers (e.g., `tzone://Microsoft/Custom`) which fail strict `TZID` normalization and validation ([home-assistant/core#165674](https://github.com/home-assistant/core/issues/165674)).
- **Missing VTIMEZONE**: Calendars may reference a `TZID` for an event but fail to include the required `VTIMEZONE` component definition, breaking time resolution ([home-assistant/core#142164](https://github.com/home-assistant/core/issues/142164)).

To prevent an entire calendar from failing to load due to a single non-compliant event, a compatibility mode is essential.

## Detailed Analysis of Broken Patterns

The `ical.compat` module handles several specific non-compliant patterns:

### DTSTART vs UNTIL Type Mismatch
As mentioned, Google Calendar and others sometimes output `DTSTART` as a `DATE-TIME` and `UNTIL` as a `DATE`.
* **Fixup (`dtstart_until_compat`)**: Detects this mismatch and automatically coerces `UNTIL` to match the type of `DTSTART`.

### Same-Day DTEND
Some providers represent all-day events with a `DTEND` that falls on the same day as `DTSTART`. According to RFC 5545, `DTEND` for `DATE` events is non-inclusive, meaning a one-day event starting on Jan 1st should have a `DTEND` of Jan 2nd.
* **Fixup (`same_day_dtend_compat`)**: Shifts the `DTEND` forward by one day to properly represent the event duration.

### Invalid Windows Timezones
Microsoft Exchange and Outlook often output Windows-specific timezone names (e.g., "Eastern Standard Time") or entirely omit `VTIMEZONE` blocks for these timezones.
* **Fixup (`timezone_compat`)**: Uses heuristics to map these proprietary `TZID` strings to standard IANA timezones and gracefully handles missing definitions.

### Invalid Date Formats
Some feeds contain date strings with an illegal time suffix (e.g., `20240115T000000` for a `DATE` type).
* **Fixup (`date_compat`)**: Strips the invalid time suffix to parse the date correctly.

## The `ical.compat` Design & PRODID Detection

Instead of using a blanket "lenient mode" that disables all validation, `ical.compat` uses targeted fixups driven by the `PRODID` (Product Identifier) of the calendar. This is superior because:
1. It applies only the necessary fixes for the known quirks of a specific provider.
2. It prevents a fixup for one provider from inadvertently masking bugs in another.

The compatibility layer relies on thread-safe context variables (`contextvars.ContextVar`) and a context manager (`contextlib.contextmanager`) to enable these fixups during parsing without modifying global state. This design ensures that parsing operations in asynchronous environments (like `asyncio`) remain safe and isolated.

## Usage: Strict Parsing vs Compatibility Mode

By default, the library operates in strict mode.

```python
from ical.calendar import Calendar
from ical.calendar_stream import IcsCalendarStream

# Strict parsing (default)
calendar = IcsCalendarStream.calendar_from_ics(ics_content)
```

To enable the compatibility layer, use the `ical.compat.enable_compat_mode()` context manager. This will detect the `PRODID` during parsing and apply the appropriate context variables for the fixups.

```python
from ical.calendar_stream import IcsCalendarStream
from ical.compat import enable_compat_mode

# Compat mode enabled
with enable_compat_mode():
    calendar = IcsCalendarStream.calendar_from_ics(ics_content)
```

## Comparison with Other Engines

Other calendar parsing engines handle these issues differently:
- `icalendar`: Tends to be more lenient by default, sometimes accepting malformed data without complaint, which can lead to unexpected behavior later in the pipeline.
- `ical`: Adheres strictly to the RFC by default, ensuring predictable data structures, while providing an opt-in, structured compatibility mechanism for real-world feeds.

## How to Contribute

If you encounter a new feed that fails to parse, you can contribute a new fixup by following these steps:
1. Analyze the specific RFC violation in the ICS data.
2. Create a new context variable in `ical/compat/` for the fixup.
3. Update `make_compat.py` to enable the fixup based on the `PRODID`.
4. Integrate the fixup check into the relevant parsing component (e.g., `event.py`, `timezoneinfo.py`).
5. Add comprehensive tests demonstrating both the strict failure and the compat success.
